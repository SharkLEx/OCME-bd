# 029 — Copy DeFi: Patterns para WEbdEX

> **Ideia central:** Copy de DeFi precisa superar dois obstáculos: ceticismo ("mais um protocolo") e confusão ("não entendo como funciona"). Os patterns abaixo atacam os dois.

---

## Os 5 Obstáculos do Leitor DeFi

```
1. "Já perdi dinheiro em DeFi antes"
2. "Não entendo como funciona de verdade"
3. "Parece coisa de quem entende de código"
4. "Onde está o catch?"
5. "Como eu sei que é real?"
```

**Cada peça de copy precisa endereçar pelo menos um desses.**

---

## Framework: DADOS → MECANISMO → PROVA

```
DADOS:     "76% de assertividade histórica"
MECANISMO: "via arbitragem triangular em pools de liquidez Polygon"
PROVA:     "Verifique on-chain: 0x6995077c49d920D8..."
```

Este framework funciona porque:
- **Dados** capturam atenção (número concreto)
- **Mecanismo** explica COMO (sem mistério)
- **Prova** elimina ceticismo (verificável, não promessa)

---

## Patterns de Headline

### Pattern 1: Número + Verbo + Contexto
```
"$1.58M gerenciados on-chain. Todo ciclo de 21h."
"76% de assertividade. 18 meses de dados verificáveis."
"369.369.369 BD em supply fixo. Para sempre."
```

### Pattern 2: Pergunta Retórica + Resposta Direta
```
"O que é uma subconta? Capital seu. Código nosso. Nenhum intermediário."
"Como funciona o ciclo 21h? Arbitragem → registro on-chain → relatório → repete."
"Por que non-custodial importa? Porque seu capital não deveria depender de nossa sobrevivência."
```

### Pattern 3: Contraste (Mercado vs WEbdEX)
```
"DeFi genérico: liquidez anônima, sem transparência.
WEbdEX: subconta individual, ciclo verificável, dados públicos."
```

### Pattern 4: Antes/Depois Técnico
```
"Antes: você não sabia o que acontecia com seu capital.
Depois: relatório on-chain a cada 21h. Verificável por qualquer um."
```

## Erros de Copy para Evitar

| Erro | Por quê | Alternativa |
|------|---------|-------------|
| "Rendimento de X%" | Promessa regulatória | "Assertividade histórica de X%" |
| "Seguro" | Toda DeFi tem risco | "Non-custodial, verificável" |
| "Fácil de usar" | Clichê vazio | "Interface no Telegram — sem instalação" |
| "Revolucionário" | Hype sem substância | "Arbitragem triangular on-chain desde 2023" |
| "Melhor do mercado" | Não verificável | "76-78% de assertividade. Verifique on-chain." |

---

## Copy para bdZinho (ensinar o padrão)

O bdZinho deve usar estes patterns organicamente:
1. **Sempre que citar dados** → incluir fonte (on-chain / banco / ciclo de data)
2. **Sempre que explicar mecanismo** → ir de simples para técnico
3. **Nunca prometer resultado futuro** → sempre historical data
4. **Sempre oferecer verificação** → link explorer, "/stats", etc.
