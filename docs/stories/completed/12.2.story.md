## 🎯 Goal

Adicionar Function Calling (tool use) ao bdZinho — permitindo que ele consulte dados on-chain em tempo real durante a conversa, com circuit breaker, timeout por tool, fallback padronizado, e rate limiting por usuário.

---

## 📋 Context

Hoje o bdZinho responde com dados estáticos (último cache do monitor engine). Com tool use:
- Usuário pergunta "qual meu portfolio?" → bdZinho chama `get_user_portfolio(wallet)` → OpenAI recebe dados frescos → resposta precisa
- Usuário pergunta "como está o mercado?" → bdZinho chama `get_market_context()` → resposta contextualizada com dados de agora

**Containers afetados:**
- `ocme-monitor` — bot Telegram (webdex_ai.py)
- `orchestrator-discord` — bot Discord (webdex_ai.py compartilhado ou cópia)

**Smith CRITICAL #2 (incorporado nos ACs):** Circuit breaker (3 falhas → disable 5min por tool), timeout individual, fallback response padronizado se tool falhar.

### AC2 — Circuit Breaker (Smith CRITICAL #2)
- [ ] Classe `ToolCircuitBreaker` em `webdex_tools.py`:
  ```python
  class ToolCircuitBreaker:
      # Threshold: 3 falhas consecutivas → OPEN (desabilitado por 5 min)
      # Após 5 min: HALF-OPEN (tenta 1 vez → CLOSED se OK, OPEN se falhar)
  ```
- [ ] Estado por tool name: CLOSED | OPEN | HALF-OPEN
- [ ] `is_available(tool_name) -> bool` — verifica se tool pode ser chamada
- [ ] `record_success(tool_name)` e `record_failure(tool_name)` — atualiza estado
- [ ] Log de warning quando circuit breaker abre: `"[tools] Circuit breaker OPEN para {tool_name}"`

### AC3 — Timeout e Fallback
- [ ] Cada tool tem timeout configurável (default: 8 segundos):
  ```python
  TOOL_TIMEOUTS = {
      "get_protocol_metrics": 8,
      "get_user_portfolio": 10,
      "get_market_context": 5,
  }
  ```
- [ ] Se tool excede timeout → lança `ToolTimeoutError`, circuit breaker registra falha
- [ ] Fallback response padronizado quando tool falha (seja timeout, circuit breaker, ou erro):
  ```
  "Não consegui buscar {dado} agora (serviço temporariamente indisponível).
   Posso te dar a última informação conhecida, ou tente novamente em alguns minutos."
  ```

### AC4 — Rate Limiting por Usuário
- [ ] Rate limit: máx 20 tool calls por hora por chat_id (qualquer combinação de tools)
- [ ] Implementar com dict em memória: `_rate_limits: dict[int, list[float]]` (timestamps das calls)
- [ ] Se rate limit excedido: retornar mensagem educada sem chamar a tool
  ```
  "Você atingiu o limite de consultas por hora (20). Aguarde alguns minutos."
  ```
- [ ] Rate limit reseta a cada hora (sliding window)

## 🛠️ Dev Notes

**OpenAI Function Calling:**
```python
# Request com tools
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=TOOLS,
    tool_choice="auto",  # "none" para desabilitar, "required" para forçar
)

# Verificar se há tool calls
if response.choices[0].finish_reason == "tool_calls":
    tool_calls = response.choices[0].message.tool_calls
    for tc in tool_calls:
        func_name = tc.function.name
        args = json.loads(tc.function.arguments)
        result = execute_tool(func_name, args)
        messages.append({
            "role": "tool",
            "tool_call_id": tc.id,
            "content": result
        })
```

**Threading:** `execute_tool` pode ser chamado em threads diferentes (telebot). Circuit breaker deve ter lock interno.

**Dados reais das tools:**
- `get_protocol_metrics` → reusar funções já existentes em `webdex_workers.py` ou consultar DB SQLite
- `get_user_portfolio` → query na tabela de usuários/wallets existente
- `get_market_context` → gas price via RPC Polygon + dados do monitor engine
