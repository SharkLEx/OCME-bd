## 🎯 Goal

Criar listener automático para eventos `SubscriptionCreated` do contrato `WEbdEXSubscription v1.1.0` na Polygon, ativando automaticamente o tier Pro no PostgreSQL quando pagamento on-chain for detectado — sem intervenção manual.

---

## 📋 Context

**Contrato:** `WEbdEXSubscription v1.1.0` — `0x6481d77f95b654F89A1C8D993654d5f877fe6E22` (Polygon)

Hoje a ativação é manual: Alex verifica o contrato e ativa no banco. Esta story automatiza completamente o fluxo:

```
Usuário paga USDC on-chain
        ↓
Evento SubscriptionCreated emitido
        ↓
Worker detecta evento (polling Web3)
        ↓
Ativa tier Pro no PostgreSQL (idempotente)
        ↓
Notificação automática Telegram + Discord
```

---

## 📦 Acceptance Criteria

### AC1 — Tabela `subscriptions` no PostgreSQL
- [ ] Criar tabela:
  ```sql
  CREATE TABLE subscriptions (
    id BIGSERIAL PRIMARY KEY,
    wallet_address VARCHAR(42) NOT NULL,
    chat_id BIGINT,  -- NULL se não conectou wallet ao bot ainda
    tier VARCHAR(20) NOT NULL,  -- 'free' | 'pro' | 'institutional'
    status VARCHAR(20) NOT NULL DEFAULT 'active',
    activated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMPTZ,
    tx_hash VARCHAR(66) NOT NULL UNIQUE,  -- idempotência por tx_hash
    log_index INT NOT NULL,
    metadata JSONB DEFAULT '{}'
  );
  CREATE UNIQUE INDEX idx_sub_tx ON subscriptions(tx_hash, log_index);
  CREATE INDEX idx_sub_wallet ON subscriptions(wallet_address);
  CREATE INDEX idx_sub_chat ON subscriptions(chat_id);
  ```
- [ ] Migration: `packages/monitor-engine/migrations/002_subscriptions.sql`

### AC2 — Worker `subscription_worker.py`
- [ ] Polling de eventos `SubscriptionCreated` a cada 60 segundos
- [ ] Busca eventos desde o último bloco processado (checkpoint em `config` table: `sub_last_block`)
- [ ] Para cada evento: chama `_process_subscription_event(event)`
- [ ] Worker iniciado em thread separada no boot do `ocme-monitor`

### AC3 — Idempotência (Smith HIGH)
- [ ] `tx_hash + log_index` como chave única — INSERT OR IGNORE se já processado
- [ ] Log de info quando evento duplicado detectado (não erro, esperado)
- [ ] Test: processar mesmo evento 3x → apenas 1 registro no banco

### AC4 — Race condition guard (Smith HIGH)
- [ ] Lock por wallet_address durante processamento:
  ```python
  _sub_processing_lock = {}  # wallet → threading.Lock()
  ```
- [ ] Timeout de 30s no lock (evitar deadlock se worker travar)
- [ ] Log de warning se lock timeout

### AC5 — Ativação no banco e notificação
- [ ] Ao detectar evento válido:
  1. Inserir em `subscriptions` (idempotente)
  2. Atualizar `users` table: `subscription_tier = 'pro'`, `subscription_expires = expires_at`
  3. Enviar mensagem Telegram ao chat_id associado (se existir):
     ```
     🎉 *Subscription Pro ativada!*
     Sua assinatura WEbdEX Pro foi ativada automaticamente.
     Válida até: {data}
     Use /start para ver todas as funcionalidades Pro!
     ```
  4. Log no canal Discord `#webdex-conquistas` via webhook

### AC6 — Verificação de tier em handlers existentes
- [ ] Função `check_subscription_tier(chat_id) -> str`:
  - Consulta `subscriptions` table pelo chat_id
  - Retorna: `'pro'` se ativo e não expirado, `'free'` caso contrário
  - Cache local de 5 minutos (evitar query a cada mensagem)
- [ ] Integrar verificação nos handlers Telegram que terão rate limit por tier (Story 14.2 usará isso)

### AC7 — Deploy
- [ ] `subscription_worker` iniciado no boot do `ocme-monitor`
- [ ] Smoke test: simular evento no Sepolia/testnet e verificar ativação
- [ ] Fallback: se contrato não responder em 30s, log error e retry na próxima iteração
