## 🎯 Goal

Migrar a memória do bdZinho de `config` table (SQLite, deque serializado) para PostgreSQL com tabela `ai_conversations`, adicionando suporte a histórico ilimitado por usuário, busca por data, e compliance LGPD com comando de deleção.

---

## 📋 Context

Hoje o bdZinho usa:
- `_ia_mem = deque(maxlen=12)` — memória em RAM, perde ao reiniciar
- `config` table no SQLite: `ai_mem_{chat_id}` = JSON serializado da deque
- Limite real: ~12 mensagens de contexto ativo

O PostgreSQL (`orchestrator-postgres`, já rodando no VPS) será a nova fonte de verdade:
- Histórico completo por usuário (90+ dias)
- `deque(20)` mantida em RAM para contexto imediato (performance)
- PostgreSQL consultado ao inicializar contexto de um usuário (cold start)

**Containers afetados:**
- `ocme-monitor` — bot Telegram (webdex_ai.py)
- `orchestrator-discord` — bot Discord (webdex_ai.py compartilhado ou cópia)

## 📦 Acceptance Criteria

### AC1 — Schema PostgreSQL
- [ ] Criar tabela `ai_conversations` no PostgreSQL:
  ```sql
  CREATE TABLE ai_conversations (
    id BIGSERIAL PRIMARY KEY,
    chat_id BIGINT NOT NULL,
    platform VARCHAR(20) NOT NULL DEFAULT 'telegram',  -- 'telegram' | 'discord'
    role VARCHAR(20) NOT NULL,  -- 'user' | 'assistant' | 'system'
    content TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    metadata JSONB DEFAULT '{}'
  );
  CREATE INDEX idx_ai_conv_chat_created ON ai_conversations(chat_id, created_at DESC);
  ```
- [ ] Migration script em `packages/monitor-engine/migrations/001_ai_conversations.sql`
- [ ] Script testado manualmente contra `orchestrator-postgres` antes do deploy

### AC2 — Migração dos dados existentes (Smith HIGH #2)
- [ ] Script `migrate_ai_memory.py` que:
  1. Lê todos os `ai_mem_{chat_id}` do SQLite
  2. Deserializa o JSON (lista de dicts `{role, content}`)
  3. Inserta em `ai_conversations` com `created_at = NOW()` (sem timestamp histórico disponível)
  4. Faz backup do SQLite antes de alterar: `cp webdex_v5_final.db webdex_v5_final.db.bak`
- [ ] Script de rollback: `rollback_ai_memory.py` que restaura SQLite do .bak se necessário
- [ ] Migration testada em ambiente local com dados reais antes do deploy no VPS
- [ ] **NUNCA** deletar dados do SQLite durante migração — apenas copiar para PostgreSQL

### AC3 — Módulo `webdex_ai_memory.py`
- [ ] Criar `packages/monitor-engine/webdex_ai_memory.py` com classe `AIMemoryManager`:
  ```python
  class AIMemoryManager:
      def __init__(self, pg_conn, chat_id, platform='telegram'):
          self._pg = pg_conn
          self.chat_id = chat_id
          self.platform = platform
          self._cache = deque(maxlen=20)  # contexto imediato

      def load_context(self, n=20) -> list[dict]:
          """Carrega últimas N msgs do PostgreSQL para cache."""

      def append(self, role: str, content: str) -> None:
          """Salva no PostgreSQL E no cache local."""

      def get_context(self) -> list[dict]:
          """Retorna cache local (deque) para passar à OpenAI."""

      def delete_all(self, chat_id: int) -> int:
          """LGPD: deleta todas as mensagens do usuário. Retorna count."""
  ```
- [ ] `load_context()` chamado no início de cada sessão (cold start)
- [ ] `append()` escreve no PostgreSQL de forma assíncrona (não bloqueia resposta)
- [ ] Conexão PostgreSQL via variável de ambiente `DATABASE_URL` (já configurada)

### AC4 — Integração em `webdex_ai.py`
- [ ] Substituir `_ia_mem` (deque global) por instâncias de `AIMemoryManager` por chat_id
- [ ] `_mem_manager = {}` — dict chat_id → AIMemoryManager (instância criada no first message)
- [ ] Histórico carregado do PostgreSQL quando usuário envia primeira mensagem da sessão
- [ ] Contexto enviado para OpenAI vem de `_mem_manager[chat_id].get_context()`

### AC5 — Comando LGPD `/esquecer_tudo`
- [ ] Novo handler Telegram: `/esquecer_tudo`
  - Deleta todas as mensagens de `ai_conversations` do chat_id
  - Limpa cache local do `AIMemoryManager`
  - Limpa entrada `ai_mem_{chat_id}` no SQLite (compatibilidade)
  - Resposta: "🗑️ Sua memória foi completamente apagada. bdZinho não lembrará de nenhuma conversa anterior."
- [ ] Confirmação antes de deletar: "⚠️ Isso apagará TODA sua memória. Confirme com /confirmar_esquecimento"

### AC6 — Graceful degradation
- [ ] Se PostgreSQL estiver indisponível: usar apenas deque(20) em RAM (sem crash)
- [ ] Log de warning quando operar sem PostgreSQL
- [ ] `load_context()` retorna lista vazia se PostgreSQL falhar (cold start silencioso)

### AC7 — Deploy
- [ ] Variável `DATABASE_URL` verificada em ambos os containers antes do deploy
- [ ] Migration SQL executada antes de fazer deploy do código Python
- [ ] Script de migração rodado no VPS com supervisão (não automaticamente)
- [ ] Smoke test após deploy: enviar mensagem ao bot → confirmar que salva no PostgreSQL

## 🛠️ Dev Notes

**PostgreSQL connection string:**
```
DATABASE_URL=postgresql://user:pass@orchestrator-postgres:5432/webdex
```
(variável já existe no container, verificar antes de assumir)

**Cuidado com threading:**
- `webdex_ai.py` roda em threads (telebot). `psycopg2` com connection pool thread-safe: usar `psycopg2.pool.ThreadedConnectionPool`
- Não usar uma única connection compartilhada (race condition)

**Padrão de escrita assíncrona:**
- Inserção em `ai_conversations` pode ser feita em thread separada para não bloquear resposta do bot:
  ```python
  threading.Thread(target=mem_manager.append, args=(role, content), daemon=True).start()
  ```

**Não usar pgvector nesta story** — embeddings são da Story 12.5. Apenas TEXT por agora.
