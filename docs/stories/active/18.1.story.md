## 🎯 Goal

1. Subir container `litellm-proxy` no Docker do VPS (mesmo network do orchestrator)
2. Migrar `webdex_ai.py` para usar LiteLLM SDK em vez de OpenAI SDK direto
3. Fallback automático: se GPT-4o-mini falha → DeepSeek V3 → Groq Llama

Zero downtime: migration é transparente para o usuário, mesma qualidade de resposta.

---

## 📋 Context

**Stack atual:**
- `webdex_ai.py` usa `openai.ChatCompletion.create(model='gpt-4o', ...)`
- Single provider, sem fallback

**Stack alvo:**
- LiteLLM proxy como intermediário
- `webdex_ai.py` usa `litellm.completion(model='gpt-4o-mini', ...)` ou OpenAI SDK apontando para proxy

**Model stack:**
| Prioridade | Modelo | Custo | Quando usar |
|------------|--------|-------|-------------|
| 1 (primary) | `openai/gpt-4o-mini` | $0.15/1M in | Padrão Free tier |
| 2 (premium) | `openai/gpt-4o` | $5.00/1M in | Tier Pro + queries complexas |
| 3 (fallback 1) | `deepseek/deepseek-chat` | $0.07/1M in | OpenAI rate limit / downtime |
| 4 (fallback 2) | `groq/llama-3.1-8b-instant` | $0.05/1M in | Speed fallback |

## 🛠️ Dev Notes

**Por que OpenAI SDK com base_url em vez de litellm lib?**
LiteLLM proxy é compatível com OpenAI API v1. Usar `OpenAI(base_url=...)` é mais simples que instalar `litellm` Python SDK (que traz 50+ dependências). O proxy faz o trabalho pesado de fallback.

**DeepSeek API:** `https://api.deepseek.com/v1` é OpenAI-compatible. `deepseek-chat` = DeepSeek-V3 (melhor custo-benefício em Jan 2026 benchmarks).

**Groq:** Free tier generoso (14.400 req/dia). `llama-3.1-8b-instant` é rápido (~200 tok/s). Usado como last-resort fallback de velocidade.

**Migração zero-downtime:** `LITELLM_API_BASE` ausente = OpenAI direto. Setar a variável no container ativa o proxy. Rollback = remover a variável.
